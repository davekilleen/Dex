# PM GitHub Setup Guide

From Aakash Gupta's guide on building a PM GitHub that gets you hired.

## Why It Matters

- Only 24% of PM candidates have a GitHub
- Every hiring manager Gupta interviewed said: if there's a GitHub linked, they check it
- A strong GitHub signals: you build things, you understand technical teams, you navigate engineering tools
- PMs placed at OpenAI, Anthropic, Meta AI all had GitHubs

## The 6 Elements That Work

### 1. One Flagship Project That Solves a Real Problem
Don't scatter across dozens of repos. Build one major project. The problem it solves should be painful and specific.

### 2. Crystal Clear Positioning
Bio format: `AI PM building [specific thing]. Shipping [type of projects].`
Pin your best 2-3 repos. Visitors should know in 5 seconds what you build and why.

### 3. Consistent Contribution Activity
Recent activity beats total stars. Shows you're actively building with latest tools.

### 4. Strategic Forks and Contributions
Fork and contribute to repos in the ecosystem where hiring managers work. Contributing to a company's own repos gets you noticed.

### 5. Documentation That Translates
Every repo needs: clear problem statements, step-by-step setup, working examples, architecture decisions explained, known limitations.

### 6. The Portfolio Effect
Each repo is a case study. Together they tell a story: this person understands products deeply enough to build them.

## 3-Week Build Plan

### Week 1: Setup
- Day 1: Create account (professional username), fill bio, create profile README, install Cursor/Claude Code
- Day 2: Build first repo — pick a real problem, use AI to generate code, commit and push

### Week 2: Flagship Project
- Pick one PM pain point and build a solution
- Types that work: PM automation tools, AI agent templates, evaluation frameworks, integration examples
- Goal: something 10 other PMs would actually use

### Week 3: Documentation and Polish
- README = your interview with the hiring manager
- Key sections: The Problem, The Solution, Tradeoffs and Decisions, What I Learned
- Add screenshots, demo GIF, example outputs
- Pin best repos, add GitHub link to LinkedIn and resume

## What to Avoid

1. **Copy-Paste GitHub** — AI-generated with no design decisions. Change at least one decision and document why.
2. **Joke Projects** — Every project should solve a problem someone would pay to have solved.
3. **Green Square Farming** — Empty commits with no real changes. Hiring managers click the squares.
4. **Engineer Cosplay** — Don't write custom algorithms. Focus on problem statement, UX, documentation.
5. **Private Repo Graveyard** — Rebuild internal tools as generalized public repos.
6. **One-and-Done** — Build one small thing per week. Consistency compounds.

## README Template (per project)

1. **The Problem** — What pain point, who has it, why it matters
2. **The Solution** — What it does, how the user interacts with it
3. **Tradeoffs and Decisions** — 1-2 choices you made and why (this separates PM from engineer GitHub)
4. **What I Learned** — Real insights, not generic "learned about APIs"
